## 3. 前端技术选型

### 3.1 前端语言：TypeScript

前端统一使用 TypeScript。

主要作用：

- 为组件属性、页面数据和业务状态提供类型约束；
- 减少接口字段变化导致的运行时错误；
- 提高重构、代码导航和多人协作的可靠性；
- 为后续前后端接口类型生成保留空间。

约定：

- 新增前端业务代码不使用 JavaScript 文件；
- 开启严格类型检查；
- 核心业务逻辑不使用无约束的 `any`；
- 公共类型放在 `frontend/src/types/`；
- 功能私有类型保留在对应 `features/` 功能目录内。

### 3.2 前端框架：Next.js App Router

前端框架使用 Next.js App Router。

主要作用：

- 管理页面、布局和路由；
- 支持服务端与客户端组件；
- 支持公开页面的 SEO 和分享；
- 支持复杂交互页面和后续功能扩展；
- 为加载、错误、权限等页面状态提供统一组织方式。

约定：

- 页面路由和布局放在 `frontend/src/app/`；
- 具体业务功能放在 `frontend/src/features/`；
- 通用组件放在 `frontend/src/components/`；
- Next.js 主要作为前端页面框架使用；
- 正式业务逻辑和数据处理由 Go 后端承担；
- 是否使用特定 Next.js 服务端能力，在对应功能设计时再决定。

### 3.3 样式方案：Tailwind CSS + CSS Variables

样式方案保持现有方向，不作调整。

- Tailwind CSS 负责页面布局、响应式和组件样式组合；
- CSS Variables 负责颜色、字体、间距、圆角、阴影等设计变量；
- 通用样式放在 `frontend/src/styles/`；
- 页面和功能模块优先复用统一设计变量；
- 不提前确定具体组件库，等实际页面设计时再评估。

### 3.4 动画方案：Motion for React + CSS

动画方案保持现有方向，不作调整。

- 简单的悬浮、颜色、透明度和状态变化优先使用 CSS；
- 复杂编排、页面切换和交互动画使用 Motion for React；
- 动画必须尊重用户的减少动态效果设置；
- 不在没有明确交互目标时增加装饰性动画。

### 3.5 前端分层

前端采用路由入口、功能模块和公共能力分离的结构。当前完整目录、目录职责和代码放置规则以 [`frontend/README.md`](../frontend/README.md) 为权威来源。

技术层面的核心约束：

- `app` 负责组合页面，不堆积具体业务逻辑；
- `features` 可以使用通用组件、服务和工具；
- `components/ui` 不依赖具体业务功能；
- 功能专属组件、类型和逻辑优先保留在对应 `features` 目录；
- `services` 只处理通信和数据适配，不承载页面展示逻辑。

## 4. 后端技术选型

### 4.1 后端语言：Go

后端统一使用 Go。

主要作用：

- 提供 HTTP API；
- 承载业务规则和数据处理；
- 访问 MySQL 和 Redis；
- 支持后续并发任务和外部能力接入；
- 通过清晰的包边界控制系统复杂度。

约定：

- 使用 Go Modules 管理依赖；
- 业务代码放在 `backend/internal/`；
- 服务启动入口放在 `backend/cmd/server/`；
- 不把业务逻辑直接写在路由和 Handler 中；
- 不因功能增加而直接拆分微服务；
- 是否需要独立 Worker、消息队列等能力，在具体功能迭代时再判断。

### 4.2 后端框架：Gin

HTTP 框架使用 Gin。

主要作用：

- 注册和分组 HTTP 路由；
- 解析请求参数；
- 输出统一响应；
- 组织认证、日志、恢复、限流等中间件；
- 为后续 REST API 提供稳定入口。

约定：

- 路由定义放在 `internal/transport/http/router/`；
- Handler 放在 `internal/transport/http/handler/`；
- 中间件放在 `internal/transport/http/middleware/`；
- Handler 只负责协议转换、参数校验和响应输出；
- 业务流程调用 `internal/application/`；
- Gin 的 `Context` 不进入领域层和数据访问层。

### 4.3 数据库：MySQL

MySQL 作为主要持久化数据库。

计划承载的基础数据类型包括：

- 用户与权限数据；
- 后续确定的业务实体；
- 内容和状态数据；
- 业务操作记录；
- 需要长期保存的关系数据。

当前只确定数据库类型，不提前确定 ORM、SQL 生成工具和 Migration 工具。具体功能进入数据库设计时，根据查询复杂度和团队维护成本再选择。

基本原则：

- 表结构由业务实体和查询场景决定；
- 使用事务保证相关写操作的一致性；
- 通过索引和唯一约束保证查询效率及数据正确性；
- 数据库变更必须保留可追踪的 Migration；
- 不把 Redis 当成 MySQL 的替代品；
- 不在功能尚未确定时提前创建大量空表。

### 4.4 缓存：Redis

Redis 作为缓存和短期状态基础设施。

可能用于：

- 热点数据缓存；
- Session 或短期登录状态；
- 验证码和一次性 Token；
- 接口限流；
- 短期任务状态；
- 需要过期时间的临时数据。

以上用途不代表全部都需要立即实现。每个功能进入迭代时，需要明确是否真的需要 Redis。

基本原则：

- MySQL 保存需要长期存在的事实数据；
- Redis 数据允许过期或丢失时重建；
- Cache Key 必须具有统一命名空间；
- 缓存必须设置合理过期时间；
- 需要考虑缓存穿透、击穿和数据一致性；
- 不把核心业务唯一数据只保存在 Redis 中。

### 4.5 后端分层

后端采用 transport、application、domain、infrastructure 分层，并由 bootstrap 在最外层完成依赖组装。当前完整目录、目录职责和代码放置规则以 [`backend/README.md`](../backend/README.md) 为权威来源。

编译期依赖方向：

```text
transport → application → domain
infrastructure ───────→ domain
bootstrap → transport / application / infrastructure
```

- `domain` 不依赖 Gin、MySQL 或 Redis；
- `application` 负责一个完整业务用例的流程编排；
- `transport` 把 HTTP 请求转换为应用层输入；
- `infrastructure` 实现领域层定义的数据访问和外部能力接口；
- `bootstrap` 负责创建依赖并完成应用组装。

## 5. 数据访问边界

前端不直接访问 MySQL 或 Redis。

```text
浏览器
→ Next.js 前端
→ Gin API
→ Application / Domain
→ MySQL 或 Redis
```

后续每个功能在开始开发前需要单独确定：

- 需要哪些 API；
- 需要哪些 MySQL 表和索引；
- 是否确实需要 Redis；
- 数据一致性和缓存失效方式；
- 对应测试和修改记录。

## 7. 技术基线结论

当前技术基线为：

> 前端使用 TypeScript、Next.js App Router、Tailwind CSS、CSS Variables、Motion for React 和 CSS；后端使用 Go 和 Gin；MySQL 负责长期业务数据，Redis 负责缓存和短期状态。项目采用单仓库、前后端分目录和后端分层结构，其他技术在具体功能迭代时再决定。
