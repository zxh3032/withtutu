# docs 目录总览

> 本文件记录 `docs/` 目录的整体结构、各文档用途和维护规则。新增、删除、移动或重命名文档时，必须同步更新本文件。

## 目录结构

```text
docs/
├── README.md
├── 技术选型说明.md
└── 迭代流程记录.md
```

## 文件说明

| 文件 | 用途 | 何时更新 |
|---|---|---|
| `README.md` | 记录文档目录结构和每份文档的职责 | 文档新增、删除、移动或改名时 |
| `技术选型说明.md` | 记录已经确认的前端、后端、数据库和缓存技术基线 | 技术选型被明确调整时 |
| `迭代流程记录.md` | 第 1～4 节维护当前有效的迭代规范，第 5～6 节记录当前迭代和历史过程 | 迭代规则变化，或开始、推进、验收一次迭代时 |

## 内容边界

`docs/` 用于存放跨前后端、需要长期维护的项目文档，例如：

- 技术选型；
- 迭代流程；
- 后续经过确认的跨端方案；
- 需要前后端共同遵守的设计说明。

以下内容不放在本目录：

- 前端具体代码修改记录，放在 `frontend/change-records/`；
- 后端具体代码修改记录，放在 `backend/change-records/`；
- 临时调试输出、构建产物和本地私人笔记；
- 尚未进入迭代、没有确认的功能设想。

## 文档类型与表达规则

- 技术说明、需求说明、架构说明和目录总览等正式正文只保留当前有效的结论、结构、约束和执行方式；
- `迭代流程记录.md` 第 1～4 节属于正式正文，第 5～6 节以及前后端修改记录、单独的讨论或决策记录属于过程记录，可以保存问题背景、分析方案、备选项、用户反馈和决策理由；
- 需要保存详细分析或聊天原文时，使用名称明确的单独过程记录文档，不将其混入正式正文；
- 过程记录应使用能够独立理解的结构化表达，写清问题、方案、决定、理由和影响，避免使用依赖当时聊天上下文的指代；
- 规则变化时直接更新正式正文并清理旧说法；过程记录中的历史信息可以继续保留；
- 过程记录中跨迭代持续有效、会影响后续开发的最终结论需要同步到对应正式正文；仅对当前迭代有效的信息保留在过程记录中；
- 各类内容的唯一权威来源以根目录 `AGENTS.md` 中的“权威来源与摘要”为准；其他正式正文可以保留带有明确来源的简要摘要，不独立维护第二套详细规则。

## 维护规则

1. 新增文档前先确认它是否需要跨迭代长期维护；
2. 文档名称应准确表达内容，不使用“临时”“新建文档”等模糊名称；
3. 文档内容发生职责变化时，同时更新本文件的文件说明；
4. 删除文档前检查是否仍被 `README.md`、`AGENTS.md` 或迭代记录引用；
5. 目录结构变化必须在同一次修改中更新本文件；
6. 具体迭代结果应写入迭代记录，不在多个文档重复维护；
7. 正式正文修改完成后搜索旧名称、旧规则和冲突表述，确保正文只呈现当前有效内容；过程记录中的历史信息不在清理范围内。
