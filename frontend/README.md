# frontend 目录总览

> 本文件记录前端目录的整体结构、各文件夹职责和代码放置规则。前端目录结构发生变化时，必须同步更新本文件。

## 技术基线

以下为前端技术摘要，完整技术选型以 [`docs/技术选型说明.md`](../docs/技术选型说明.md) 为准。

- 语言：TypeScript；
- 框架：Next.js App Router；
- 样式：Tailwind CSS + CSS Variables；
- 动画：Motion for React + CSS。

## 目录说明

| 路径 | 内容与职责 |
|---|---|
| `README.md` | 当前前端目录的结构总览和代码放置规则 |
| `change-records/` | 前端代码、页面、交互和业务逻辑修改记录 |
| `public/` | 不经过模块导入、需要直接访问的静态资源 |
| `src/app/` | Next.js 路由、布局、页面入口及路由级状态文件 |
| `src/components/ui/` | 不依赖具体业务的基础 UI 组件 |
| `src/components/shared/` | 多个功能共同使用的组合组件 |
| `src/features/` | 按实际业务功能组织页面片段、逻辑、类型和功能私有组件 |
| `src/hooks/` | 跨功能复用的 React Hooks |
| `src/lib/` | 无明确业务归属的基础工具、常量和适配能力 |
| `src/services/` | Go 后端 API 调用、请求封装和数据适配 |
| `src/stores/` | 确实需要跨页面共享的客户端状态 |
| `src/styles/` | 全局样式、Tailwind 入口和 CSS Variables |
| `src/types/` | 跨功能复用的公共 TypeScript 类型 |
| `tests/e2e/` | 从用户视角验证完整操作流程的端到端测试 |

## 代码放置规则

- 页面文件负责路由组合，不堆积大量业务逻辑；
- 一个功能的组件、逻辑和类型优先放在同一个 `features/<feature-name>/` 下；
- 只有被多个功能稳定复用的内容才提升到公共目录；
- `components/ui` 不得依赖 `features`；
- `services` 负责通信和数据转换，不负责页面展示；
- `stores` 不作为所有状态的默认存放位置，组件局部状态保留在组件内；
- 新功能进入迭代后再创建对应功能目录，不提前创建空模块。

## 修改记录规则

前端发生源代码、配置、脚本、测试、页面、交互或业务逻辑变化时，在 `change-records/` 中记录：

```text
YYYYMMDD_功能变动.md
```

同一迭代、同一功能只维护一份记录，文件名使用首次修改日期；跨日期继续修改时，在原记录中追加带时间的内容。独立功能或范围发生实质变化时创建新记录，并与相关记录相互链接。

每份记录至少包含修改时间、修改原因、修改范围、涉及文件、实际变更、逻辑变化、数据库或缓存变化、验证结果、已知风险与后续事项。没有数据库或缓存变化时明确写“无”。

## 维护规则

- 新增、删除、移动或重命名一级、二级目录时更新本文件；
- 目录职责发生变化时更新“目录说明”；
- 新增构建、测试或配置目录后补充其用途；
- 实际代码文件无需逐个罗列，但关键入口文件和配置文件应在本文件中说明；
- 目录总览更新应与结构变更在同一次迭代中完成。
