# withtutu 大模型协作规范

本文约束后续使用大模型或自动化编码工具修改 `withtutu` 项目时的工作方式。所有进入仓库工作的 Agent 都应先阅读本文。

## 1. 项目基线

- 项目采用正常迭代流程，不使用 MVP、V1.1、V2 划分最终功能；
- 每次只实现已经确认进入当前迭代的功能；
- 不根据长期设想提前创建大量功能、接口、数据表或基础设施；
- 当前技术基线以 `docs/技术选型说明.md` 为准；
- 当前迭代和历史记录以 `docs/迭代流程记录.md` 为准。

## 2. 已确认技术选择

- 前端语言：TypeScript；
- 前端框架：Next.js App Router；
- 样式：Tailwind CSS + CSS Variables；
- 动画：Motion for React + CSS；
- 后端语言：Go；
- 后端框架：Gin；
- 数据库：MySQL；
- 缓存：Redis。

未经用户明确确认，不得替换以上技术选择，也不得自行增加会改变整体架构的框架或基础设施。

## 3. 工作前检查

开始修改前必须：

1. 阅读 `README.md`；
2. 阅读目标目录的总览文件：`docs/README.md`、`frontend/README.md` 或 `backend/README.md`；
3. 阅读 `docs/技术选型说明.md`；
4. 阅读 `docs/迭代流程记录.md` 中的当前迭代；
5. 检查工作区状态，保留用户已有修改；
6. 确认修改属于当前迭代范围；
7. 确认应该修改前端、后端还是两者。

如果请求超出当前迭代，但用户明确要求开始新功能，应先在迭代记录中新增一条迭代，再进行开发。

## 4. 目录总览维护

以下目录各自使用 `README.md` 记录整体结构和内容职责：

- `docs/README.md`；
- `frontend/README.md`；
- `backend/README.md`。

新增、删除、移动或重命名对应目录中的文件夹时，必须在同一次修改中更新相应的 `README.md`。新增关键入口、配置或长期维护文件时，也需要在目录总览中补充说明。

## 5. 前端分层规则

- 路由、页面和布局放入 `frontend/src/app/`；
- 功能代码放入 `frontend/src/features/`；
- 通用基础组件放入 `frontend/src/components/ui/`；
- 跨功能业务无关组件放入 `frontend/src/components/shared/`；
- 后端通信和数据适配放入 `frontend/src/services/`；
- 公共 Hooks、工具、状态、样式和类型进入对应目录；
- 不在页面文件中堆积大段业务逻辑；
- 不让通用 UI 组件反向依赖具体业务功能。

## 6. 后端分层规则

后端依赖方向为：

```text
transport → application → domain
                       ↑
              infrastructure
```

- Gin 路由、Handler 和中间件放在 `internal/transport/http/`；
- Handler 只负责协议转换、输入校验和响应输出；
- 用例和业务流程编排放在 `internal/application/`；
- 领域实体、规则和接口放在 `internal/domain/`；
- MySQL 和 Redis 的实现放在 `internal/infrastructure/`；
- Gin Context 不得进入 application、domain 或 infrastructure；
- 业务逻辑不得直接写在路由或数据库实现中；
- 不因单个功能新增而自行拆分微服务。

## 7. 修改记录要求

任何实际代码或逻辑变更，都必须在对应目录记录：

- 前端：`frontend/change-records/`；
- 后端：`backend/change-records/`。

文件名统一使用：

```text
YYYYMMDD_功能变动.md
```

中文描述中的标准形式为：

```text
年月日_功能变动.md
```

示例：

```text
20260713_登录功能调整.md
```

同一天、同一功能的连续修改更新同一个记录文件；不同功能创建不同文件。

每份修改记录至少包含：

- 修改时间；
- 修改原因；
- 修改范围；
- 涉及文件；
- 实际代码变更；
- 逻辑变化；
- 数据库或缓存变化；
- 验证结果；
- 已知风险与后续事项。

如果一次修改同时影响前后端，需要分别创建前端和后端记录，并相互链接。

## 8. 迭代记录要求

- 开始新功能前，在 `docs/迭代流程记录.md` 新增迭代；
- 开发中及时更新状态、范围变化和关键决策；
- 开发完成后补充实际变更、测试结果和修改记录链接；
- 未通过验证时不得把状态标记为“已完成”；
- 新需求不应偷偷并入当前迭代，必须明确记录范围变化或新建迭代。

## 9. 代码与数据变更原则

- 优先做满足当前功能的最小完整设计；
- 不实现没有进入当前迭代的附加功能；
- 不创建没有实际用途的抽象层；
- 新增依赖前说明用途，并确认现有能力无法合理完成；
- MySQL 结构变化必须提供可追踪 Migration；
- Redis Key 必须具有统一命名空间和过期策略；
- 不把必须长期保存的唯一数据只放在 Redis；
- 不在前端保存数据库凭证、Redis 凭证或后端密钥；
- 不提交真实密钥和用户敏感数据。

## 10. 验证与交付

完成修改后必须：

1. 运行与修改范围相匹配的格式、类型和测试检查；
2. 检查是否误改无关文件；
3. 更新前端或后端修改记录；
4. 如目录结构发生变化，更新目标目录的 `README.md`；
5. 更新当前迭代的实际变更和验证结果；
6. 向用户说明完成内容、验证情况和仍待决定的事项。

不允许只修改代码而不保留变更记录。
