# withtutu 大模型协作规范

本文约束后续使用大模型或自动化编码工具修改 `withtutu` 项目时的工作方式。所有进入仓库工作的 Agent 都应先阅读本文。

## 1. 项目基线

- 每次只实现已经确认进入当前迭代的功能；
- 不根据长期设想提前创建大量功能、接口、数据表或基础设施；
- 当前技术基线以 `docs/技术选型说明.md` 为准；
- 当前迭代和历史记录以 `docs/迭代流程记录.md` 为准。

## 2. 已确认技术选择

以下内容是便于执行的技术基线摘要，完整说明和唯一权威来源为 `docs/技术选型说明.md`：

- 前端语言：TypeScript；
- 前端框架：Next.js App Router；
- 样式：Tailwind CSS + CSS Variables；
- 动画：Motion for React + CSS；
- 后端语言：Go；
- 后端框架：Gin；
- 数据库：MySQL；
- 缓存：Redis。

未经用户明确确认，不得替换以上技术选择，也不得自行增加会改变整体架构的框架或基础设施。摘要与技术选型说明不一致时，以技术选型说明为准，并在同一次修改中同步修正摘要。

## 3. 工作前检查

开始任务前必须完成以下基础检查：

1. 阅读并遵守当前 `AGENTS.md`；
2. 检查 Git 分支和工作区状态，识别并保留用户已有修改；
3. 阅读 `docs/迭代流程记录.md` 中的当前迭代，确认任务是否已经进入范围；
4. 确认本次任务影响文档、前端、后端还是多个范围。

其他文档按照任务需要阅读，不要求在每次连续修改中机械重复：

- 第一次接触项目，或任务涉及产品定位、整体结构和项目状态时，阅读根目录 `README.md`；
- 首次进入目标目录、判断代码放置位置或调整目录结构时，阅读对应的 `docs/README.md`、`frontend/README.md` 或 `backend/README.md`；
- 任务涉及源代码、依赖、架构、数据库或缓存时，阅读 `docs/技术选型说明.md`；
- 需要理解历史决策或追溯变更原因时，再阅读相关历史迭代和修改记录；
- 同一个连续任务中，已经读取且没有发生变化的文档无需重复读取。

如果请求超出当前迭代，但用户明确要求开始新功能，应先在迭代记录中新增一条迭代，再进行开发。

## 4. 文档维护规则

### 4.1 正文与过程记录分离

- 根目录和各目录的 `README.md`、`AGENTS.md`、技术说明、需求说明、架构说明以及其他作为当前依据的文档属于正式正文，只呈现当前有效的结论、结构和执行规则；
- `docs/迭代流程记录.md` 的“迭代原则”“标准迭代流程”“迭代状态”和“迭代记录模板”属于正式正文，“当前迭代”和“历史迭代”属于过程记录；
- 前后端 `change-records/` 和明确标记为讨论记录、决策记录或聊天归档的文档属于过程记录，可以保存问题背景、分析方案、备选项、用户反馈、决策理由和必要的聊天原文；
- 前后端修改记录可以记录与实际改动直接相关的讨论摘要，但应写清具体问题、候选方案、最终决定、选择理由和影响，不使用“刚才讨论过”“按上面的方案”等依赖聊天上下文的表达；
- 详细分析或聊天原文确有长期保留价值时，应放入单独且名称明确的过程记录文档，不混入正式正文；过程记录放入 `docs/` 时，需要同步登记到 `docs/README.md`；
- 过程记录中跨迭代持续有效、会影响后续开发的最终结论，必须同步写入对应正式正文；仅对当前迭代有效的信息不需要提升为长期规则。后续执行以正式正文为准，过程记录只用于追溯上下文。

### 4.2 权威来源与摘要

| 内容 | 唯一权威来源 |
|---|---|
| 项目定位、整体结构和项目状态 | 根目录 `README.md` |
| 大模型与自动化工具协作规则 | 根目录 `AGENTS.md` |
| 技术选型 | `docs/技术选型说明.md` |
| 迭代规则、当前迭代和历史记录 | `docs/迭代流程记录.md` |
| 前端目录职责和代码放置规则 | `frontend/README.md` |
| 后端目录职责、代码放置规则和依赖方向 | `backend/README.md` |

- 其他正式正文可以保留便于阅读的简要摘要，但必须链接或明确指向权威来源，不得在摘要中独立扩展出第二套规则；
- 摘要与权威来源发生冲突时，以权威来源为准，并在同一次修改中同步修正摘要；
- 同一事项需要补充细节时，优先修改权威来源，其他文档只更新受影响的摘要或链接。

### 4.3 正文维护

- 需求或规则发生变化时，直接重写正式正文并清理失效表述，不在正文中追加“原来如何”“删除了什么”“从什么改成什么”等变更痕迹；
- 正文描述优先使用当前态表达，例如写“项目按明确功能逐项迭代”，不要通过罗列已经放弃的名称来解释当前规则；
- 只有兼容性、数据迁移、安全审计或回滚确实依赖历史信息时，正式正文才保留必要背景，并明确它对当前实现仍然存在的影响；
- 完成正式正文修改前，应在正文文档范围内搜索旧名称、旧规则和冲突表述，确保同一事项只保留一套当前有效说法；过程记录中的历史信息不要求清除。

### 4.4 目录总览维护

以下目录各自使用 `README.md` 记录整体结构和内容职责：

- `docs/README.md`；
- `frontend/README.md`；
- `backend/README.md`。

新增、删除、移动或重命名对应目录中的文件夹时，必须在同一次修改中更新相应的 `README.md`。新增关键入口、配置或长期维护文件时，也需要在目录总览中补充说明。

## 5. 前端分层规则

前端目录职责和代码放置规则的详细权威来源为 `frontend/README.md`。以下内容是所有 Agent 必须遵守的执行摘要：

- `frontend/src/app/` 只放 Next.js 路由、布局、页面入口和路由级状态文件，页面负责组合功能，不堆积大段业务逻辑；
- 一个功能的组件、逻辑、Hooks 和私有类型优先放入同一个 `frontend/src/features/<feature-name>/`，避免按文件类型拆散到多个目录；
- 不依赖具体业务的基础 UI 组件放入 `frontend/src/components/ui/`；
- 被多个功能稳定复用的组合组件放入 `frontend/src/components/shared/`，只被一个功能使用的组件不要提前提升为共享组件；
- 后端通信、请求封装和数据适配放入 `frontend/src/services/`，不在服务层处理页面展示逻辑；
- 跨功能复用的 Hooks、工具、状态、样式和类型分别进入现有对应目录；局部状态和功能私有能力继续留在功能内部；
- `app` 可以组合 `features` 和公共组件，`features` 可以依赖公共组件、服务和基础工具；公共组件、服务和基础工具不得反向依赖 `app` 或具体 `features`；
- 新功能必须沿用上述分层，不得在局部功能中另建一套平行架构；
- 不是每个小功能都必须创建完整目录和抽象层，能在现有功能内清晰完成时不要额外拆层。

## 6. 后端分层规则

后端目录职责、代码放置规则和依赖方向的详细权威来源为 `backend/README.md`。以下内容是所有 Agent 必须遵守的执行摘要。

后端编译期依赖方向为：

```text
transport → application → domain
infrastructure ───────→ domain
bootstrap → transport / application / infrastructure
```

- Gin 路由、Handler 和中间件放在 `internal/transport/http/`；Handler 只负责协议转换、输入校验、调用应用用例和响应输出；
- 用例、业务流程编排和事务边界放在 `internal/application/`，不得依赖 Gin 或具体 MySQL、Redis 实现；
- 领域实体、核心业务规则以及确有需要的外部能力接口放在 `internal/domain/`，不得依赖 transport、application 或 infrastructure；
- MySQL 和 Redis 的具体实现放在 `internal/infrastructure/`，通过实现领域接口为应用层提供能力；
- `internal/bootstrap/` 是组装入口，可以创建并连接各层的具体依赖，但不得承载业务规则；
- Gin Context 不得进入 application、domain 或 infrastructure；
- 业务逻辑不得直接写在路由、Handler、数据库或缓存实现中；
- `internal/shared/` 只存放真正跨层复用且与具体业务无关的基础能力，不得成为无法归类代码的默认位置；
- 新功能必须沿用现有依赖方向和目录职责，前后功能的组织方式应保持一致，不得为单个功能另建平行分层；
- 不是每个功能都必须机械创建 transport、application、domain 和 infrastructure 的全部文件；没有相应职责时不创建空层或只做转发的包装层；
- 不因单个功能新增而自行拆分微服务。

## 7. 修改记录要求

前端或后端发生源代码、配置、脚本、测试、接口、数据结构、缓存或业务逻辑变更时，都必须在对应目录记录：

- 前端：`frontend/change-records/`；
- 后端：`backend/change-records/`。

文件名统一使用：

```text
YYYYMMDD_功能变动.md
```

中文描述中的标准形式为：

```text
年月日_功能变动.md
```

示例：

```text
20260713_登录功能调整.md
```

同一迭代、同一功能只维护一份修改记录，文件名使用首次修改日期。跨日期继续修改时，在原记录中追加带时间的变更内容；独立功能或范围发生实质变化时，创建新记录并相互链接。

每份修改记录至少包含：

- 修改时间；
- 修改原因；
- 修改范围；
- 涉及文件；
- 实际变更；
- 逻辑变化；
- 数据库或缓存变化；
- 验证结果；
- 已知风险与后续事项。

需要保留与代码改动直接相关的讨论时，可以增加“方案与决策”部分，按照“问题背景、候选方案、最终决定、选择理由、实际影响”记录。详细聊天原文应放入单独的过程记录文档，不直接堆入修改记录。

如果一次修改同时影响前后端，需要分别创建前端和后端记录，并相互链接。

## 8. 迭代记录要求

- 开始新功能前，在 `docs/迭代流程记录.md` 新增迭代；
- 开发中及时更新状态、范围变化和关键决策；
- 开发完成后补充实际变更、测试结果和修改记录链接；
- 开发和验证结束后将状态标记为“待验收”；
- 只有用户明确确认验收通过后才能标记为“已完成”；是否已经合并 Git 分支不作为默认完成条件，除非本次验收标准明确要求合并；
- 未通过验证时不得标记为“待验收”或“已完成”；
- 新需求不应偷偷并入当前迭代，必须明确记录范围变化或新建迭代。

## 9. 代码与数据变更原则

### 9.1 代码修改确认

- 在修改任何源代码、配置、脚本、测试、Migration 或依赖文件前，必须先向用户说明准备修改的内容、预计涉及的文件和主要影响，并取得明确确认；
- 用户只要求分析、诊断、评审、查看或提出方案时，只能进行只读检查，不得顺带修改代码；
- 一次确认覆盖已经说明的范围，范围内不需要逐文件重复询问；如果过程中发现必须扩大范围、改变方案或增加依赖，应停止修改并重新确认；
- 用户没有明确确认时，可以继续做只读调查、解释和方案设计，但不得写入代码文件。

### 9.2 代码删除

- 删除现有代码前，必须说明删除范围、替代实现和影响，并检查调用、导入、配置、测试及数据兼容性；
- 删除前应确认存在清晰、可追溯的恢复点，例如当前分支已有提交或为本次删除建立独立检查点；不得通过覆盖用户未提交修改来制造恢复点；
- 删除范围获得用户确认且恢复点明确后，可以在当前功能分支直接删除，不强制先保留大段注释代码；
- 只有确实需要短期对照、灰度切换或兼容过渡时，才临时注释或禁用代码，并使用中文说明原因、替代实现和清理条件；
- 注释、禁用或删除后都必须通过格式、类型、编译和相关测试检查，最终交付不得保留无用途的死代码；
- 删除整个文件、公共接口、数据库字段或对外行为时，除上述流程外，还必须说明引用范围、兼容性和回滚影响。

### 9.3 中文注释

- 新代码首先通过清晰命名、直接结构和合理组织表达意图，不要求为能够直接理解的语句机械添加注释；
- 公共接口、核心流程、复杂条件、异常处理、数据转换和不直观的实现必须重点注释；
- 关键中文注释应使后续阅读者能够理解功能目的、业务规则、边界条件以及采用当前设计的原因；
- 注释应解释“为什么这样做”和“需要满足什么约束”，不逐行翻译代码语法，也不使用大量无信息量注释干扰阅读；
- 修改代码时同步检查相关注释，禁止保留与实际行为不一致的过期说明。

### 9.4 简单实现与最小改动

- 优先做满足当前功能的最小完整设计；
- 优先在现有结构中完成局部修改，避免为了小功能进行大范围重构；
- 函数和方法应保持职责清晰，但不要仅为了缩短行数或追求形式上的“单一职责”而拆成大量只能跳转阅读的小函数；
- 只有出现稳定复用、明确边界、替换需求或真实测试隔离需求时才增加抽象；禁止为单一实现、单一调用或尚未出现的未来需求提前创建多层接口、工厂、管理器、适配器或包装层；
- 能直接、清晰表达的逻辑不要引入设计模式、通用框架或复杂泛型；若确实需要复杂方案，必须先说明简单方案无法满足的具体原因；
- 修改范围必须与用户确认的功能直接相关，不借机整理或重构无关代码；
- 不实现没有进入当前迭代的附加功能；
- 不创建没有实际用途的抽象层；
- 新增依赖前说明用途，并确认现有能力无法合理完成；

### 9.5 数据与安全

- MySQL 结构变化必须提供可追踪 Migration；
- Redis Key 必须具有统一命名空间和过期策略；
- 不把必须长期保存的唯一数据只放在 Redis；
- 不在前端保存数据库凭证、Redis 凭证或后端密钥；
- 不提交真实密钥和用户敏感数据。

## 10. 验证与交付

完成修改后必须：

1. 运行与修改范围相匹配的格式、类型和测试检查；
2. 检查是否误改无关文件；
3. 检查新增中文注释是否准确，并列出仍处于注释或禁用状态的待删除代码；
4. 更新前端或后端修改记录；
5. 如目录结构发生变化，更新目标目录的 `README.md`；
6. 更新当前迭代的实际变更和验证结果；
7. 向用户说明完成内容、验证情况和仍待决定的事项。

不允许只修改代码而不保留变更记录。
